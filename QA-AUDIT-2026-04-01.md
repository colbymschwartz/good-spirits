## 2. Security Findings

### SEC-01: API Key Passed in URL Query String
- **Severity:** MEDIUM (dormant — currently no key configured)
- **File:** `src/utils/ocrService.ts:32`
- **Confidence:** 95%
- **Finding:** `fetch(\`${VISION_API_URL}?key=${API_KEY}\`)` — API keys in URLs are logged by proxies, CDNs, crash reporters, and browser dev tools
- **Risk:** When the Vision API key is eventually configured, it will be exposed in any network monitoring. Google Cloud Vision keys can incur charges.
- **Fix:** Use request headers instead: `headers: { 'X-goog-api-key': API_KEY }`. Or better, proxy through a backend that holds the key.

### SEC-02: No Input Validation on URL Import
- **Severity:** LOW
- **File:** `src/components/ImportModal.tsx:207-228`
- **Confidence:** 85%
- **Finding:** `extractRecipeFromUrl(urlInput.trim())` fetches any user-provided URL with no validation. Could be used to probe internal network addresses (SSRF-like), though React Native's fetch is sandboxed.
- **Risk:** Minimal on mobile — React Native fetch is sandboxed and can't access `file://` or localhost services easily. But good hygiene to validate.
- **Fix:** Add basic URL validation: ensure it starts with `https://`, reject private IP ranges.

### SEC-03: Silent Storage Write Failures
- **Severity:** MEDIUM
- **File:** `src/store/storage.ts:17`
- **Confidence:** 95%
- **Finding:** `async set(key: string, value: unknown): Promise<void> { try { ... } catch {} }` — completely swallowed errors
- **Risk:** If AsyncStorage hits its size limit (~6MB on some devices) or has a corruption issue, user data silently stops saving. Users lose custom cocktails, favorites, and notes without knowing.
- **Fix:** At minimum, log the error. Ideally, surface a toast/alert when storage fails.

### SEC-04: Dependency Vulnerabilities
- **Severity:** LOW
- **File:** `package-lock.json`
- **Confidence:** 90%
- **Finding:** `npm audit` reports 5 vulnerabilities (2 moderate, 3 high) in transitive deps: `node-forge` (DoS + signature forgery), `picomatch` (ReDoS), `yaml` (stack overflow)
- **Risk:** These are in dev/build tooling dependencies, not runtime code shipped to users. No direct exploitation path in the app.
- **Fix:** Run `npm audit fix` to clean them up. Low urgency.

### SEC-05: Permission Descriptions Are Good ✅
- **File:** `app.json`
- **Finding:** Camera and photo permissions have clear, user-friendly descriptions explaining why they're needed. `ITSAppUsesNonExemptEncryption: false` is correctly set. No issues here.

### SEC-06: No Hardcoded Secrets ✅
- **Finding:** Grep for secrets/tokens/passwords found only the properly env-var-sourced Vision API key (`process.env.EXPO_PUBLIC_GOOGLE_CLOUD_VISION_API_KEY || ''`). EAS project ID in `app.json` is not a secret. No `.env` files committed. Clean.

## 3. Code Quality Findings

### Architecture: Clean and Well-Organized ✅

The codebase follows a solid, conventional structure:

```
src/
  components/    # Reusable UI (modals, cards, bars)
  screens/       # Screen-level components
  navigation/    # Tab + stack navigators
  store/         # Zustand store + AsyncStorage wrapper
  data/          # Static cocktail/ingredient/technique data
  theme/         # Colors, typography, spacing tokens
  types/         # TypeScript interfaces
  utils/         # Utility functions
```

**Strengths:**
- Clean separation of concerns
- Zustand store is well-designed with proper selectors
- Theme system is consistent and well-organized
- Type definitions are thorough

### CQ-01: CocktailsScreen is Getting Complex
- **Severity:** LOW
- **File:** `src/screens/CocktailsScreen.tsx` (677 lines)
- **Confidence:** 85%
- **Finding:** The filtering/search logic (lines ~130-285) is a large `useMemo` with nested conditionals and multiple passes over the data. It works, but it's getting hard to reason about.
- **Impact:** Maintainability. Not a bug risk today, but will be harder to modify as features are added.
- **Suggestion:** Extract the filtering logic into a custom hook like `useFilteredCocktails()` or a utility function. This would also make it independently testable.

### CQ-02: CocktailDetailScreen is Large but Acceptable
- **Severity:** LOW  
- **File:** `src/screens/CocktailDetailScreen.tsx` (862 lines)
- **Confidence:** 80%
- **Finding:** 862 lines including styles. The component itself is feature-rich (photos, notes, share, batch, remix) but each feature is relatively self-contained.
- **Suggestion:** Could extract `StarRating` into its own component file (it's already a separate function). The note-save debounce logic could be a custom hook. But these are "nice to have" refactors.

### CQ-03: Duplicated `parseIngredientId` Function
- **Severity:** LOW
- **File:** `src/components/CreateCocktailModal.tsx`, `src/components/ImportModal.tsx`, `src/components/RemixModal.tsx`
- **Confidence:** 95%
- **Finding:** The exact same `parseIngredientId()` function is copy-pasted across three files.
- **Fix:** Move to `src/utils/ingredientUtils.ts` and import.

### CQ-04: Duplicated `generateId` Function
- **Severity:** LOW
- **File:** `src/components/CreateCocktailModal.tsx`, `src/components/ImportModal.tsx`
- **Confidence:** 95%
- **Finding:** Same `generateId()` implementation in two files.
- **Fix:** Move to utils.

### CQ-05: Untyped Navigation (`useNavigation<any>`)
- **Severity:** LOW
- **File:** `src/screens/MyBarScreen.tsx:83`, `src/screens/HistoryScreen.tsx:27`, `src/screens/FavoritesScreen.tsx:47`
- **Confidence:** 95%
- **Finding:** Three screens use `useNavigation<any>()` instead of properly typed navigation props. TypeScript can't catch navigation bugs in these screens.
- **Fix:** Define navigation types for each stack (like `CocktailsStack` already does) and use them.

### CQ-06: Duplicate `parseRecipeText` Implementations
- **Severity:** LOW
- **File:** `src/components/ImportModal.tsx` and `src/utils/recipeExtraction.ts`
- **Confidence:** 90%
- **Finding:** Two different implementations of recipe text parsing. `ImportModal` has its own inline `parseRecipeText()` and also imports `extractRecipeFromUrl` from `recipeExtraction.ts` which has another `parseRecipeText()`. The URL flow uses the utility version; the text-paste flow uses the inline version. They have slightly different logic.
- **Risk:** Bug divergence — fixing a parsing bug in one won't fix it in the other.
- **Fix:** Consolidate into a single implementation in `recipeExtraction.ts`.

### CQ-07: Dead Code — `extractRecipeFromImage`
- **Severity:** LOW
- **File:** `src/utils/recipeExtraction.ts:41-45`
- **Confidence:** 95%
- **Finding:** `extractRecipeFromImage()` is a stub that just `console.log`s and returns null. It's never called — OCR is handled by `ocrService.ts` instead. Has a TODO comment.
- **Fix:** Remove it, or redirect to `ocrService.ts`.

### CQ-08: `PlaceholderScreen` Uses `any` Type
- **Severity:** LOW
- **File:** `src/screens/PlaceholderScreen.tsx:7`
- **Confidence:** 95%
- **Finding:** `({ route }: any)` — but this screen appears to be unused/placeholder. Low concern.

### CQ-09: ArtDecoDivider Uses `any` for Style Prop
- **Severity:** LOW
- **File:** `src/components/ArtDecoDivider.tsx:6`

## 5. Performance Assessment

### Good Practices Already in Place ✅
- `FlatList` with `maxToRenderPerBatch={15}` and `windowSize={10}` — proper virtualization
- `React.memo` on `CocktailCard` — prevents unnecessary re-renders
- `useMemo` for filtered results and bar match data — expensive computation cached
- `useCallback` for handlers and render functions — referential stability for FlatList
- Native driver for animations (`useNativeDriver: true`) — offloads to UI thread

### PERF-01: Search Runs Filtering Logic Twice
- **Severity:** LOW
- **File:** `src/screens/CocktailsScreen.tsx` (~lines 175 and 225)
- **Confidence:** 85%
- **Finding:** When searching, the `filtered` memo first filters all cocktails by keywords, then runs a second pass to expand matching variations. The keyword splitting and matching happens twice.
- **Impact:** Negligible with ~120 cocktails. Would matter at 1000+.
- **Suggestion:** Combine into a single pass if the list grows significantly.

### PERF-02: Photo URIs in State
- **Severity:** LOW
- **Confidence:** 80%
- **Finding:** Photo URIs (file paths to images) are stored in Zustand state and AsyncStorage. The URIs themselves are fine, but if a user adds many photos, the state object grows and every state update causes the entire photos record to be serialized.
- **Impact:** Unlikely to matter with <200 photos. Just something to watch.

### No Memory Leak Concerns Detected ✅
- No interval/timeout leaks (timers are properly managed in CocktailDetailScreen)
- No event listener leaks
- No unclean async operations
